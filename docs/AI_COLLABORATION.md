## 1. AI 협업 방법론

Lecture Pulse는 기획 단계부터 배포까지 전 과정에서 AI를 파트너로 활용했다. 단순히 코드 자동완성에 그치지 않고, 설계 결정, 알고리즘 구현, 아키텍처 선택까지 AI와의 대화를 통해 검증하고 발전시켰다.

### 협업 단계별 활용 방식

```
┌─────────────────────────────────────────────────────────────────┐
│  기획 단계        │  설계 단계        │  개발 단계       │  검증 단계  │
│                   │                   │                  │             │
│  • 문제 정의 검증 │  • DB 스키마 설계 │  • 알고리즘 구현 │  • 엣지 케이스 │
│  • 페르소나 분석  │  • API 명세 설계  │  • 프롬프트 튜닝  │  • 코드 리뷰 │
│  • 경쟁 서비스    │  • 실시간 아키텍처│  • 폴백 로직 설계 │  • 성능 분석 │
│    분석           │    결정           │                  │             │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 프롬프트 엔지니어링 전략

AI 기능을 개발하면서 프롬프트의 품질이 결과물의 품질을 직접 결정한다는 것을 반복해서 확인했다. 다음 3가지 전략을 체계적으로 적용했다.

### 전략 1: 한국어 응답 명시 지정

**문제**: AI에게 영어로 프롬프트를 작성하면 영어로 응답하는 경향이 있다. 한국 대학 수업 데이터를 처리할 때 영어 응답은 무의미하다.

**해결**: 모든 프롬프트 끝에 "한국어로 작성해주세요" 또는 프롬프트 자체를 한국어로 작성.

```typescript
// 클러스터링 프롬프트 — 한국어로 작성하여 한국어 질문 처리 최적화
const prompt = `아래는 수업 중 학생들이 보낸 질문 목록입니다. 의미가 비슷한 질문끼리 군집으로 묶어주세요.
같은 주제나 개념에 대한 질문이면 표현이 달라도 같은 군집으로 묶으세요.
...`;
```

**결과**: 한국어 조사, 어미 변형, 축약어를 올바르게 처리하는 클러스터링 결과를 얻었다.

### 전략 2: JSON 포맷 강제

**문제**: AI는 기본적으로 자연어로 응답한다. 코드에서 파싱해야 하는 경우 마크다운 코드 블록, 설명 텍스트, 불규칙한 포맷이 섞여 들어오면 파싱 에러가 발생한다.

**해결**: 응답 형식을 프롬프트에 명시하고, "반드시 JSON만 출력, 다른 텍스트 없이" 라는 강한 제약을 부여. 그리고 방어적 파싱(JSON 부분만 정규식으로 추출)을 추가.

```typescript
// 프롬프트에 포맷 명시
`응답 형식 (반드시 JSON만 출력, 다른 텍스트 없이):
{"clusters": [{"ids": ["uuid1", "uuid2"], "label": "군집 요약"}, ...]}`;

// 방어적 파싱 — 마크다운 등이 섞여도 JSON 추출
const jsonMatch = text.match(/\{[\s\S]*\}/);
if (!jsonMatch) {
  console.error("AI clustering: no JSON found in response:", text);
  return null;
}
const json = JSON.parse(jsonMatch[0]);
```

**결과**: Claude가 가끔 마크다운 코드 블록으로 감싸서 응답해도 정규식으로 안전하게 추출 가능해졌다.

### 전략 3: 역할 프레이밍 (Role Framing)

**문제**: 일반적인 질문 형태로 요청하면 AI가 범용적인 응답을 생성한다. 수업 리포트처럼 특정 맥락이 있는 경우, AI가 그 맥락을 이해하고 적합한 분석을 해야 한다.

**해결**: 프롬프트 상단에 명확한 역할과 수신자를 지정.

```typescript
// 리포트 생성 프롬프트 — 역할 프레이밍
`아래는 한 수업의 실시간 피드백 데이터입니다. 교수님을 위한 간결한 수업 리포트를 한국어로 작성해주세요.

포함할 내용:
1. 전체 이해도 요약 (한 줄)
2. 주요 혼란 구간 분석 (시간대별 변화가 있다면)
3. 학생 질문에서 드러나는 핵심 어려움 포인트
4. 다음 수업을 위한 제안 (2~3개)`;
```

**결과**: AI가 "교수에게 유용한 정보"를 선별하는 방식으로 응답하게 되어, 단순 통계 나열이 아닌 실질적 인사이트를 생성한다.

## 3. 구체적 협업 사례

### 사례 1: 질문 클러스터링 알고리즘 설계

**배경**: "수업 중 학생 질문을 묶어서 보여주면 어떨까?"라는 아이디어에서 출발했다. 처음에는 단순 키워드 매칭을 생각했지만, 한국어의 특성상 동일한 개념을 "미분이 뭐예요?", "미분의 정의가 궁금합니다", "미분 개념을 모르겠어요" 처럼 완전히 다른 표현으로 질문하는 경우가 많았다.

**AI와의 협업 과정**:

1. AI에게 "한국어 질문을 의미 기반으로 클러스터링하는 방법"을 물었고, 임베딩 기반, TF-IDF, 그리고 LLM 직접 호출 3가지 옵션을 검토했다.
2. 임베딩과 TF-IDF는 추가 라이브러리 의존성과 벡터 DB가 필요하다는 단점이 있었다. LLM 직접 호출은 간단하지만 API Key 없이는 동작하지 않는다.
3. AI가 제안: "API Key 있을 때는 LLM으로, 없을 때는 경량 키워드 오버랩 알고리즘으로 폴백하는 이중 설계를 해보세요."
4. 폴백 알고리즘의 핵심 로직(불용어 제거, 토큰 분리, partialMatch 함수)을 AI와 함께 설계했다.

**결과물** (`/api/cluster/route.ts`):
```typescript
// AI 군집화 시도 → 실패 시 키워드 폴백
const clusters = (await clusterByAI(questions)) ?? clusterByKeywords(questions);

// 한국어 부분 일치 — 조사 변형 대응
function partialMatch(a: string, b: string): boolean {
  if (a === b) return true;
  const shorter = a.length <= b.length ? a : b;
  const longer = a.length > b.length ? a : b;
  return shorter.length >= 2 && longer.includes(shorter);
}
```

**배운 점**: "조사 변형 대응"이라는 한국어 특수성을 알고리즘 설계에 반영한 것이 AI와의 대화에서 나온 인사이트였다. "미분"과 "미분의", "미분을"이 같은 토큰으로 처리되어야 한다.

### 사례 2: DB 스키마 설계

**배경**: 실시간 피드백 시스템의 DB를 설계할 때, 이해도 신호를 어떻게 저장할지가 첫 번째 고민이었다.

**선택지 A**: 학생별 현재 상태만 저장 (responses 테이블에 upsert)
- 장점: 항상 최신 상태만 있어 조회가 단순
- 단점: 시간대별 추이 분석이 불가능

**선택지 B**: 모든 신호를 append-only로 저장
- 장점: 시간대별 분석 가능, 수업 흐름 추적 가능
- 단점: 데이터가 쌓이고, 현재 상태 계산 시 집계 필요

**AI 의견**: "수업 리포트에서 '14:30에 혼란이 집중됐다'는 분석을 하려면 반드시 시간 정보가 있어야 합니다. 단기적으로 데이터가 늘어나는 것은 문제가 아니고, 세션 종료 후에는 집계 결과만 남겨도 됩니다."

**결론**: 선택지 B 채택. 모든 응답을 `created_at`과 함께 append-only로 저장. 현재 상태는 클라이언트에서 Map으로 집계.

```sql
-- 모든 신호를 시간 정보와 함께 저장
create table responses (
  id uuid primary key default gen_random_uuid(),
  session_id uuid not null references sessions(id) on delete cascade,
  student_id text not null,
  type response_type not null,
  created_at timestamptz default now()
);

-- 빠른 시간순 조회를 위한 복합 인덱스
create index idx_responses_session on responses(session_id, created_at);
```

### 사례 3: 실시간 아키텍처 결정

**배경**: 교수 대시보드에서 학생 이해도를 실시간으로 보여주려면 어떤 기술을 써야 하는가? WebSocket 직접 구현, Server-Sent Events, 폴링, 또는 외부 서비스?

**검토 옵션**:

| 방식 | 장점 | 단점 |
|------|------|------|
| 폴링 (1~2초) | 구현 단순 | 지연, 서버 부하 |
| SSE | 단방향 실시간 | 브라우저 연결 제한 |
| WebSocket 직접 구현 | 완전 제어 | 복잡한 서버 구현, Vercel 제약 |
| Supabase Realtime | DB 변경 즉시 전파, 인프라 불필요 | Supabase 의존 |

**AI 의견**: "이미 Supabase를 DB로 쓰고 있다면, Realtime은 추가 인프라 없이 DB INSERT를 WebSocket으로 브로드캐스트해줍니다. 테이블에 `alter publication supabase_realtime add table responses;` 한 줄로 실시간이 활성화됩니다."

**결론**: Supabase Realtime 채택. DB 변경이 곧 실시간 이벤트가 되는 구조로, 별도 WebSocket 서버 없이 구현.

```typescript
// 교수 대시보드 — 이해도 신호 실시간 구독
supabase
  .channel(`responses-${sessionId}`)
  .on("postgres_changes", {
    event: "INSERT",
    schema: "public",
    table: "responses",
    filter: `session_id=eq.${sessionId}`,
  }, (payload) => {
    // 새 신호 즉시 반영
    updateStudentState(payload.new.student_id, payload.new.type);
  })
  .subscribe();
```

## 4. AI 폴백 설계

API Key 없이도 핵심 기능이 동작해야 한다는 원칙을 처음부터 세웠다. 이는 데모 환경, 비용 절감 상황, 또는 API 장애 시에도 서비스가 의미 있게 동작하도록 하기 위함이다.

### 폴백 결정 흐름

```
API 호출 시도
    │
    ├── ANTHROPIC_API_KEY 없음 → 즉시 폴백 (API 호출 없이)
    │
    ├── API 호출 실패 (네트워크 오류, 4xx/5xx) → 폴백
    │
    ├── 응답 파싱 실패 (JSON 없음, 포맷 오류) → 폴백
    │
    └── 성공 → AI 결과 사용
```

### 클러스터링 폴백 알고리즘 상세

```typescript
function clusterByKeywords(questions) {
  // 1단계: 불용어 제거 + 토큰 추출
  //   "미분이 왜 이렇게 어려운가요?" → ["미분", "어려운가요"]
  //   한국어/영어 불용어 모두 처리

  // 2단계: 클러스터 간 토큰 오버랩 측정
  //   오버랩 점수 = 겹치는 토큰 수 / 최대 토큰 수
  //   임계값 0.2 이상이면 같은 클러스터

  // 3단계: 클러스터 토큰 풀 갱신
  //   새 질문이 클러스터에 합류하면 해당 클러스터의 토큰 풀에 새 토큰 추가
  //   → 클러스터가 확장될수록 더 많은 질문을 포착

  // 4단계: 단일 항목 클러스터 제거
  //   질문이 1개인 클러스터는 의미 없음 → 제외
}
```

## 5. 협업에서 배운 점

### AI는 "왜?"를 물어봐야 가장 유용하다

단순히 "이 코드 구현해줘"보다 "왜 이렇게 해야 하는지, 다른 방법과 비교해줘"라고 물었을 때 훨씬 유용한 인사이트가 나왔다. DB 스키마 설계에서 append-only 방식을 채택한 것, 클러스터링 폴백을 설계한 것 모두 "왜?"를 반복한 결과물이다.

### 프롬프트는 코드와 같다

처음에는 대충 작성한 프롬프트로 AI를 호출했다가 파싱 에러, 영어 응답, 불필요한 설명 텍스트 포함 등의 문제를 반복해서 겪었다. 프롬프트를 코드처럼 명확하고 구조적으로 작성하면 결과물의 품질이 눈에 띄게 올라간다는 것을 학습했다.

### 폴백 없는 AI 기능은 리스크다

API Key가 없거나 API가 응답하지 않을 때 앱 전체가 기능 불능이 되는 설계는 프로덕션에서 위험하다. 모든 AI 호출 지점에 폴백 로직을 설계하는 습관이 생겼다.

### 한국어 처리의 특수성을 AI에게 물어봐야 한다

한국어는 조사, 어미 변형, 합성어 등의 특성상 영어 NLP 방식을 그대로 적용하면 잘 동작하지 않는다. "이 알고리즘을 한국어에 적용하면 어떤 문제가 생길까?"라고 AI에게 물었을 때, 조사 변형을 처리하는 `partialMatch` 함수 아이디어를 얻었다.

### AI가 틀릴 수 있다 — 반드시 검증하라

AI가 제안한 Supabase Realtime 설정 코드 중 일부는 실제 Supabase SDK API와 맞지 않는 부분이 있었다. AI의 제안을 공식 문서와 대조하고 실제로 테스트하는 습관이 중요하다. AI는 강력한 파트너지만 최종 검증 책임은 개발자에게 있다.
